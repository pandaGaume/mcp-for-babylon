What if your 3D interface
could understand you?

We built an MCP server that lives inside a Babylon.js scene. Now any LLM client can inspect and command live 3D objects in real time — no plugins, no special runtime. Here is why this matters far beyond WebGL demos.

Experimental build note ● Open-source · github.com/pandaGaume/mcp-for-babylon ● Apache 2.0

There is a quiet assumption buried in almost every industrial interface ever built: the operator speaks the machine's language. They learn the symbology, the widget hierarchy, the exact click sequence that turns a valve yellow. The machine never learned theirs.

Large language models have changed that assumption at the application layer — chat is cheap. But at the interface layer — the 3D scene, the SCADA panel, the GIS map — nothing has changed. The screen still waits for the right click in the right place.

This project is a small experiment in closing that gap.

The inversion: the browser becomes the server

The Model Context Protocol (MCP) is an open standard that lets LLM clients discover and invoke tools exposed by a running server. The conventional picture is a Node.js process exposing database queries or API calls. We did something different.

We made the Babylon.js browser page itself the MCP server. Scene objects — cameras, lights, meshes — are exposed as MCP resources. Moving a camera, highlighting a mesh, dimming a light: those are MCP tools. A thin Node.js WebSocket tunnel bridges the browser to any MCP-compatible client. Claude Code, MCP Inspector, or any LLM toolchain can then talk directly to the live scene.

Claude / GPT / Copilot / MCP Inspector │ │ POST /mcp (Streamable HTTP) │ GET /sse (SSE stream) ▼ WsTunnel (Node.js relay) │ │ ws://localhost:3000/provider ▼ McpServer ← runs INSIDE the browser page Babylon.js scene

The browser never was just a display surface. This makes it a first-class participant in an agentic workflow.

The architecture: Behaviour and Adapter

The key design insight is a strict separation between what a thing is and what engine it lives in.

A Behavior owns the MCP identity of a category of objects: its namespace, its URI template, its tool schemas. It contains zero engine imports. It does not know what BabylonJS is. A camera behavior is simply the declaration that something called babylon://camera/{cameraId} exists, that it can be read, and that tools like camera_animate_to or camera_shake can be invoked on it.

An Adapter is the only layer that touches the engine. It holds a URI→object index, subscribes to scene events, serializes live state into JSON, and dispatches mutations. When a camera is added to the scene, the adapter emits an event. The server catches it, sends a notifications/resources/list_changed to the client. The client's tool list updates. All of this happens without any coordination logic in the behavior itself.

// Wire up — the adapter is the only BJS-aware component const adapter = new McpCameraAdapter(scene); const cambehavior = new McpCameraBehavior(adapter); // Register and start await McpServerBuilder .withName("Command Center") .withWsUrl("ws://localhost:3000/provider") .register(cambehavior, ... an others ) .build() .start();

State types — the data contracts returned by resources/read — are plain TypeScript interfaces with zero engine dependency. ICameraState, ILightState, the frustum types: all framework-free, all serializable anywhere. The MCP contract is cleanly decoupled from the rendering engine.

"The behavior defines what something is in the protocol. The adapter defines how to reach it in the engine. The state defines what it looks like to the outside world. Three concerns, three files, zero coupling." Design rationale

This is translation, not reasoning

Something worth naming precisely: the LLM here is not reasoning about your scene. It is translating natural language into a structured tool call that was already defined by the behavior's JSON Schema. The intelligence lives in the interface contract, not in the model.

"Highlight the pressure sensors in sector 4" becomes mesh_set_highlight({ uri: "babylon://mesh/sensor_p4_*", color: {...} }) — not because the model understands your P&ID, but because your MeshBehavior declared that tool clearly enough for the model to recognize the mapping.

That distinction matters operationally. Translation is fast, cheap, and auditable. The tool call is the receipt. You can log it, validate it, require a confirmation step. You are not handing keys to a reasoning system; you are giving a natural language front-end to a deterministic command API that you control entirely.

The screen no longer waits for the right click. It waits for intent — and translates it.

Now imagine the last-mile interfaces

The pattern is not BabylonJS-specific. The Behavior/Adapter split was designed precisely so that the protocol layer is reusable. Here is where it gets interesting.

Scenarios
Use case · Industrial
The command center that listens

An operator oversees 400 assets on a 3D plant floor. An alarm fires. Instead of navigating a widget tree, they say: "Show me all pressure readings above threshold in the east compressor hall, highlight the three highest in red, and move the camera to frame them."

Three tool calls. One sentence. The interface responds in the same frame cycle because the MCP server lives in the same browser process as the scene. No round trip to a backend. No plugin. The operator never left the screen.

Use case · SCADA / Process
Natural language as process command

A ProcessBehavior exposes valves, pumps, and flow meters as MCP resources. The adapter reads live OPC-UA tags. An LLM translates "reduce flow on loop 3 by 15% and flag the adjustment for review" into a validated, logged, confirmation-gated tool call. (with every concern with cyber and necessary human supervision).

The model is not autonomous. It proposes. The confirmation step — also a tool — is the human in the loop. The translation layer and the safety layer are orthogonal.

Use case · GIS / Cesium
Spatial queries without spatial syntax

A CesiumTerrainBehavior exposes tiles, layers, and entities as MCP resources. "Fly to the pipeline segment crossing the flood zone and toggle the risk layer." The camera flies. The layer toggles. The GIS operator never opened a filter dialog.

The coordinate system translation — WGS84 in, camera space out — lives entirely in the adapter. The behavior schema is the same camera_animate_to tool, unchanged.

Use case · Digital Twin
The interface that explains itself

A maintenance technician asks: "What changed on this asset in the last 24 hours?" The behavior's resources/read returns a rich state snapshot. The LLM compares it to a previous snapshot, generates a natural language diff, and highlights the changed components in the 3D view — all in one turn.

Building a new behavior takes an afternoon

The surface area is deliberately small. To add a new category of objects to the MCP surface, you implement two classes:

A Behavior subclass that overrides three protected methods: _buildTools(), _buildResources(), _buildTemplate(). You declare your tool schemas in JSON Schema. You define your URI template. You import nothing from any 3D engine.

An Adapter subclass that extends McpAdapterBase, implements readResourceAsync and executeToolAsync, and wires into your engine's event system to emit onResourcesChanged when the scene changes.

That is the entire contract. The server, the tunnel, the session management, the URI routing, the MCP handshake — all inherited, all invisible.

class SensorBehavior extends McpBehavior { _buildTools(): McpTool[] { return [{ name: "sensor_highlight", description: "Highlights a sensor mesh in the scene.", inputSchema: { /* JSON Schema — the only thing the LLM sees */ properties: { uri: { type: "string" }, color: { /* r,g,b */ } } } }]; } // _buildResources and _buildTemplate: similar, ~10 lines each }

What comes next

This is an experiment, not a product. Acces Rights and Cyber are still to defne. The current implementation covers cameras and lights in BabylonJS. The interesting work ahead is in the behaviors: mesh manipulation, material control, annotation overlays, data binding from live telemetry streams. Each one is a new vocabulary the interface gains.

The broader question is architectural. As 3D interfaces — game engines, digital twins, geospatial platforms — become the primary canvas for industrial and operational data, the question of how operators interact with that data becomes critical. Mouse-driven navigation was the answer for 30 years. It may not be the answer for the next 10.

The LLM is not the intelligence here. The interface contract is. The model is the parser. You write the grammar.
