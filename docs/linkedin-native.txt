What if your 3D interface could understand you?

Not "understand" in the sci-fi sense. Understand as in: you say what you need, and the interface responds.

I've been working on an experiment that makes this concrete — and I think the implications go well beyond WebGL demos.

---

🔄 The inversion

We built an MCP server that runs inside a Babylon.js browser page.

Not behind it. Not beside it. Inside it.

Scene objects — cameras, lights, meshes — are exposed as MCP resources. Moving a camera, highlighting an asset, dimming a light: those are MCP tools. Any LLM client (Claude Code, MCP Inspector, or your own) can now inspect and command a live 3D scene in real time.

The browser stopped being a display surface. It became a participant.

---

🧱 The architecture that makes it reusable

The interesting part isn't the BabylonJS integration. It's the pattern underneath it.

We separated two concerns cleanly:

▸ A Behavior owns the MCP identity of a category of objects — its namespace, URI template, tool schemas. It imports nothing from any 3D engine. It doesn't know what BabylonJS is.

▸ An Adapter is the only layer that touches the engine. It holds a live URI→object index, subscribes to scene events, serializes state, dispatches mutations.

State types — what gets returned when you read a resource — are plain TypeScript interfaces with zero engine dependency.

Three concerns. Three files. Zero coupling.

This matters because the same pattern works for any 3D or geospatial engine. The behavior layer is reusable. The adapter is the only thing you rewrite.

---

🎯 Translation, not reasoning

Something worth naming precisely.

The LLM here is not reasoning about your scene. It is translating natural language into a structured tool call that was already defined by the behavior's JSON Schema.

"Highlight the pressure sensors in sector 4" becomes sensor_highlight({ uri: "...", color: {...} }) — not because the model understands your P&ID, but because your behavior declared that tool clearly enough for the model to recognize the mapping.

That distinction matters operationally:
→ Translation is fast, cheap, auditable
→ The tool call is the receipt — you can log it, validate it, require confirmation
→ You're not handing keys to a reasoning system
→ You're giving a natural language front-end to a deterministic command API you control entirely

The LLM is the parser. You write the grammar.

🏭 Now imagine the last-mile interfaces

This is where I think it gets genuinely interesting.

INDUSTRIAL COMMAND CENTER
An operator oversees 400 assets on a 3D plant floor. An alarm fires.
Instead of navigating a widget tree: "Show me all pressure readings above threshold in the east compressor hall, highlight the three highest in red, and move the camera to frame them."
Three tool calls. One sentence. Same frame cycle — no backend round-trip.

SCADA / PROCESS CONTROL
A ProcessBehavior exposes valves, pumps, flow meters as MCP resources. The adapter reads live OPC-UA tags. "Reduce flow on loop 3 by 15% and flag for review" becomes a validated, logged, confirmation-gated tool call.
The model proposes. The human confirms. The confirmation step is just another tool.

GIS / CESIUM
A CesiumTerrainBehavior exposes tiles, layers, entities. "Fly to the pipeline segment crossing the flood zone and toggle the risk layer." Camera flies. Layer toggles. No filter dialog opened.
Coordinate system translation (WGS84 in, camera space out) lives in the adapter. The behavior schema is unchanged.

DIGITAL TWIN
A technician asks: "What changed on this asset in the last 24 hours?" The behavior returns a rich state snapshot. The LLM diffs it, narrates the changes, highlights the affected components in 3D.

---

⚙️ Building a new behavior takes an afternoon

The surface area is small by design.

You implement two classes:

1. A Behavior subclass — override _buildTools(), _buildResources(), _buildTemplate(). Declare your schemas. Define your URI template. Import nothing from any engine.

2. An Adapter subclass — implement readResourceAsync and executeToolAsync. Wire into your engine's event system.

That's the entire contract. Server, tunnel, session management, URI routing, MCP handshake: all inherited, all invisible.

💭 The bigger question

Mouse-driven navigation was the answer for 30 years.

As 3D interfaces — game engines, digital twins, geospatial platforms — become the primary canvas for industrial and operational data, the question of how operators interact with that data becomes critical.

The screen no longer has to wait for the right click. It can wait for intent — and translate it.
